# 0002 — 403 for a foreign workspace, 404 for a foreign resource

**Status:** Accepted

## Context

Harbor is multi-tenant: every row belongs to a workspace, and a teammate's API
key belongs to exactly one workspace. When a caller references something outside
their workspace, the response must not leak whether that thing exists — a status
code is an oracle if you are not careful. Two shapes of route need answering:

- **Workspace-scoped routes** carry the tenant in the path
  (`/api/workspaces/{workspaceId}/...`).
- **By-id resource routes** carry only an opaque id
  (`/api/conversations/{id}`, `/api/contacts/{id}`, …).

## Decision

Use the status code that reveals the least for each shape.

- A **workspace-scoped route** whose `{workspaceId}` is not the caller's is
  `403 Forbidden`, decided by `WorkspaceScopeFilter` **without touching the
  database**. Because no lookup happens, a foreign workspace and a workspace id
  that does not exist are indistinguishable — the 403 cannot be used to probe
  which workspaces are real. The response is identical
  (`Workspace access denied`) in both cases.
- A **by-id resource** in another workspace is `404 Not Found`. Every resource
  query is scoped to the caller's workspace (`... && c.WorkspaceId ==
  User.GetWorkspaceId()`), so a row the caller may not see simply is not found —
  identical to an id that never existed.

## Consequences

- Neither code leaks existence: 403 says "not your workspace" without a DB hit;
  404 says "no such resource for you" whether or not it exists elsewhere.
- The rule is enforced in one global filter for scoped routes and one scoping
  predicate per resource query, so new endpoints inherit it by construction.
- It can look inconsistent at a glance ("why 403 here but 404 there?"), so the
  distinction is pinned by `WorkspaceScopeMatrixTests` and documented in
  [architecture.md](../architecture.md#authentication-and-authorization).
- Writes to a foreign resource also 404 rather than 403, for the same reason.
